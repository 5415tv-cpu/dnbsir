잉크를 절약하고 인쇄 결과물이 깨끗하도록 불필요한 색상이 제거된 인쇄 전용 레이아웃입니다.

동네비서 플랫폼 핵심 정보 구조도 (Information Architecture)

1. 데이터베이스 스키마 및 관계 정의 (ERD)

동네비서 서비스의 핵심 비즈니스 모델인 가맹점(Stores), 상품/메뉴(Products), 주문내역(Orders), 택배 정보(Parcel Bookings)의 연동 관계 및 필수 필드 명세입니다. 
            erDiagram
                STORES ||--o{ PRODUCTS : "registers (1:N)"
                STORES ||--o{ ORDERS : "manages (1:N)"
                STORES ||--o{ PARCEL_BOOKINGS : "books (1:N)"
                USERS_JWT ||--o| USERS : "auth_bridge (1:1)"
                
                STORES {
                    string store_id PK "가맹점 식별 ID (휴대폰)"
                    string name "가맹점 상점명"
                    string owner_name "대표자명"
                    string membership "등급 (free/premium)"
                    int wallet_balance "지갑 잔액"
                }
                PRODUCTS {
                    int id PK "상품 고유 ID"
                    string store_id FK "가맹점 식별 ID"
                    string name "상품명/메뉴명"
                    int price "가격"
                    int sort_order "정렬 순서"
                    int is_active "노출 상태"
                    string landing_url "이동 주소"
                    int inventory "재고 수량"
                }
                ORDERS {
                    int id PK "주문 고유 ID"
                    string store_id FK "가맹점 식별 ID"
                    string item_name "상품 요약 정보"
                    int amount "결제 금액"
                    string customer_phone "주문자 연락처"
                    string payment_method "결제 수단"
                    string settlement_status "정산 상태"
                }
                PARCEL_BOOKINGS {
                    int id PK "택배 접수 ID"
                    string store_id FK "가맹점 식별 ID"
                    string receiver_name "수화인 이름"
                    string receiver_address "도로명 주소"
                    string invoice_no "운송장 번호"
                    string status "접수 상태"
                }
                USERS_JWT {
                    string phone_number PK "인증 휴대폰 번호"
                    string auth_provider "로그인 수단"
                    string role "유저 권한"
                }

2. 시스템 컴포넌트 아키텍처 및 폴백(Fallback) 흐름도

사용자 프론트엔드 인터페이스(키오스크 UI)와 API 라우터 통신 및 DB 스토리지 연동 구조도이며, 통신 단절이나 DB 장애 시 로컬 폴백 JSON으로 우회하는 예외 처리 흐름을 보여줍니다. 
            flowchart TD
                subgraph Frontend [사용자 화면 / 키오스크 UI]
                    M_UI[market.html 
 산지직송 마켓 화면]
                    S_UI[hybrid_simulator.html 
 동적 메뉴/하이브리드 시뮬레이터]
                end

                subgraph API_Routers [FastAPI 백엔드 라우터 계층]
                    M_Router[routers/market.py 
 마켓/주문/토스 결제 API]
                    S_Router[store_router.py 
 키오스크 가맹점 상품 API]
                    A_Router[routers/auth.py 
 하이브리드 인증/로그인 API]
                end

                subgraph DB_Layer [데이터베이스 스토리지 계층]
                    SQLite_DB[(database.db 
 SQLite 로컬 DB)]
                    PG_DB[(PostgreSQL 
 실전 운영 원격 DB)]
                end

                M_UI --> M_Router
                S_UI --> S_Router
                S_UI --> A_Router
                
                M_Router --> SQLite_DB
                M_Router --> PG_DB
                S_Router --> SQLite_DB
                A_Router --> SQLite_DB
                A_Router --> PG_DB
                
                S_UI -.->|네트워크 단절 / DB 장애 시| FB[안전 폴백 로컬 JSON 객체 호출]
